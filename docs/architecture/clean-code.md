# 整洁代码指南

## 常量优于魔法数字

-   用具名常量替换硬编码数值
-   使用有描述性的常量名，说明其用途
-   将常量放在文件顶部或专用常量文件中

## 有意义的命名

-   变量、函数和类应清楚表达其用途
-   命名应说明存在的原因及其用法
-   避免缩写，除非是通用缩写

## 智能注释

-   不要注释代码做了什么——让代码自解释
-   用注释说明为什么要这样做
-   为 API、复杂算法和不明显的副作用添加文档

## 单一职责

-   每个函数只做一件事
-   函数应简短且专注
-   如果需要注释解释函数做什么，应将其拆分

## DRY（不要重复自己）

-   将重复代码提取为可复用函数
-   通过合理抽象共享通用逻辑
-   保持单一事实来源

## 清晰结构

-   将相关代码放在一起
-   以逻辑层次组织代码
-   使用一致的文件和文件夹命名规范

## 封装

-   隐藏实现细节
-   暴露清晰接口
-   将嵌套条件语句提取为有意义的函数

## 代码质量维护

-   持续重构
-   及早修复技术债务
-   让代码比你发现时更整洁

## 测试

-   修复 bug 前先编写测试
-   保持测试代码可读且易维护
-   测试边界情况和错误场景

## 版本控制

-   编写清晰的提交信息
-   保持提交小而专注
-   使用有意义的分支命名
